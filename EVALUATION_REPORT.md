## 2. Strategy A — Analysis

### Strengths

- **Speed:** A single embedding call followed by an FAISS inner-product scan completes in under 10 ms on CPU.
- **Precision for exact-match queries:** When the user's vocabulary exactly matches the indexed text, cosine similarity alone is highly discriminative.
- **Determinism:** Identical queries always return identical results — valuable for caching and reproducibility.

### Weaknesses

- **Vocabulary mismatch:** A user asking "how do we prevent servers from overloading?" will miss chunks describing "rate limiting" and "token bucket" unless those words appear in the query.
- **Single embedding representation:** The query is represented by exactly one point in embedding space. If the query is ambiguous or under-specified, retrieval quality degrades.
- **No contextual expansion:** Technical acronyms (HPA, RRF, PBFT) may not expand to their full meaning in the embedding space of a general-purpose model.

---

## 3. Strategy B — Analysis

### Strengths

- **Synonym injection:** The mock Gemini expansion adds terms like "throttling", "rate cap", "sliding window" for a query about API abuse prevention — increasing recall from the sparse synonym coverage.
- **Multi-query RRF:** Running 3 query variants and fusing with RRF k=60 widens the candidate pool while suppressing noise from any single poor-quality variant.
- **HyDE mode:** Generating a hypothetical document passage and embedding it closes the vocabulary gap between conversational queries and technical indexed text.
- **Cross-encoder reranking (when enabled):** The ms-marco-MiniLM-L-6-v2 model re-scores query-chunk pairs with full attention, catching semantic matches that cosine similarity in a lower-rank space misses.

### Weaknesses

- **Latency:** 3× embedding calls + cross-encoder adds ~30–60 ms.
- **Expansion noise:** For very specific queries, adding generic synonyms can introduce irrelevant candidates. Example: expanding "Raft consensus" with "leader election" adds value; adding "eventually consistent" dilutes precision.
- **Determinism limited:** Heuristic expansions are deterministic in the mock, but in production with a live LLM there is stochasticity — complicating A/B testing.

## 4. Query-Level Analysis

### Query: "How does the system handle peak load?"

- **Strategy A** retrieves autoscaling and load balancing chunks via direct semantic similarity. Good results.
- **Strategy B** adds terms like "traffic spike", "scale out", "horizontal scaling" — improving recall for chunks that describe the mechanism without using "peak load" directly.
- **Winner: B** (higher recall)

### Query: "What happens when a node fails?"

- Both strategies retrieve fault tolerance content.
- Strategy B's expansion includes "failover", "crash recovery", "replication" — casting a wider net.
- **Winner: B** (marginally)

### Query: "How is traffic distributed across servers?"

- Strong overlap between strategies — both retrieve load balancing content effectively.
- **Winner: Tie**

### Query: "How do we prevent API abuse?"

- Strategy A may miss "throttling" and "token bucket" if the query uses "abuse" vs "limit" vocabulary.
- Strategy B explicitly adds "rate limiting", "quota", "sliding window".
- **Winner: B** (clear recall improvement)

### Query: "How does Kubernetes scale applications automatically?"

- Both retrieve HPA content; the query is specific enough for Strategy A.
- **Winner: Tie / A** (faster, same precision)

## 5. Failure Cases

### 5.1 Over-expansion

When Strategy B expands "circuit breaker" with very broad terms like "resilience" and "failure isolation", the top-K results can include chunks about connection pooling and database replication — semantically related but not specifically relevant to circuit breakers. This reduces precision.

**Mitigation:** Use `mode="synonyms"` for specific technical queries; reserve `mode="full"` for vague conversational queries.

### 5.2 HyDE semantic drift

HyDE generates a hypothetical passage and embeds it instead of the original query. For well-defined technical queries, this can move the query vector away from the correct region of embedding space if the generated passage uses different technical vocabulary.

**Mitigation:** Use HyDE selectively for under-specified queries ("tell me about scaling"), not for precise technical queries ("HPA configuration in Kubernetes").

### 5.3 BM25 sparse retrieval limitations

Hybrid search combines BM25 (exact term matching) with dense retrieval. BM25 excels for technical acronyms but fails entirely for synonymous queries. The default fusion weight (dense=0.7, BM25=0.3) prioritises dense retrieval, which is appropriate for this corpus.

## 6. Similarity Metric — Cosine vs Euclidean

### Cosine Similarity (chosen)

$$\text{sim}(\mathbf{u}, \mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \|\mathbf{v}\|}$$

After L2 normalisation, this reduces to inner product — enabling FAISS `IndexFlatIP`.

**Advantages for this corpus:**
- Chunk length varies (50–512 tokens). Cosine similarity is magnitude-invariant, so a short chunk that is highly topically relevant scores the same as a long chunk on the same topic.
- High-dimensional embedding spaces (384 dims) cause Euclidean distances to concentrate — all pairwise distances become similar, making ranking unreliable.

### Euclidean Distance (rejected)

$$d(\mathbf{u}, \mathbf{v}) = \|\mathbf{u} - \mathbf{v}\|_2$$

- Penalises longer vectors regardless of direction
- Poor discrimination in 384+ dimensions (curse of dimensionality)
- Score range `[0, ∞)` — difficult to set a universal relevance threshold

## 7. Recommendations

1. **Default to Strategy B** for production — the recall improvement outweighs the latency cost for asynchronous search contexts.
2. **Enable cross-encoder reranking** when top-K ≤ 10 and latency budget allows (+40–80 ms).
3. **Use `mode="synonyms"`** for domain expert queries (precise vocabulary), `mode="full"` for end-user conversational queries.
4. **Index tuning:** Switch from `IndexFlatIP` to `IndexIVFFlat` (nlist=100) once the corpus exceeds 50,000 chunks.
5. **Swap mock for production Vertex AI** by changing one import line — the architecture is designed for zero-diff migration.
